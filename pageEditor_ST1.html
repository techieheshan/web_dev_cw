Template & Navigation

Technical Description

template.html defines the structure every page in the site is built from: a <header> with the logo (an <img> with descriptive alt text) and a two-tier <nav> (utility bar plus main nav) with links to every page; a <main> left empty as the slot each student fills with their own content; and a <footer> with four columns (brand, page links, page-owner links, student details). The header uses position: sticky, with JavaScript (js/navbar.js) adding a header-scrolled class once the user scrolls past a threshold to animate the utility bar closed. On narrow screens a hamburger toggle button collapses the nav, using CSS order values so the logo and toggle stay together whether the menu is open or closed. The shared colour palette and the Poppins/Inter font pairing (reference 6) are defined centrally in css/global.css so every page inherits them automatically. The logo itself (reference 5) was AI-generated by Student 2 rather than sourced from a stock site.

Accessibility

All images, including the logo, have descriptive alt text. The header, nav, main, and footer use semantic HTML elements rather than generic divs, and the mobile toggle button carries aria-label and aria-expanded, kept in sync with its open/closed state by JavaScript.

Home page

Technical Description

home.html follows the shared template, then adds a full-bleed hero banner (background photo credited in reference 1) using the width: 100vw / position: relative / left: 50% / margin-left: -50vw technique to break out of the centred container, a CSS Grid of three content cards (repeat(auto-fit, minmax(280px, 1fr))) each with a photo (credited in reference 2 and reference 3), heading, description and checklist, and a focus-list of links to all four students' Content Pages. Page-specific CSS (the hero fade-in animation, card hover-lift, focus-list hover states) is embedded in a <style> block, as required for this page.

Accessibility

The hero photo and all three card images have descriptive alt text rather than filenames. The focus-list links to each teammate's Content Page are real <a> elements, keyboard reachable by default. The validator originally flagged the mission-card headings for jumping from <h1> straight to <h3>; adding a proper <h2> before them fixed the outline for screen-reader navigation.

Gallery Page

Technical Description

Each thumbnail in gallery.html is a <div class="thumb" tabindex="0" role="button"> carrying data-title, data-desc, and data-img attributes. A single hidden modal is populated by JavaScript rather than duplicated per photo. All interactivity (opening/closing the modal, hover feedback, and the colour/font toggles inside the modal) is handled with addEventListener (reference 7) rather than CSS :hover, as required. Page-specific styling and script live in their own external files, css/gallery.css and js/gallery.js. The four thumbnail photos are credited in reference 2, 3 and 4.

Accessibility

Each thumbnail carries tabindex="0", role="button", and a descriptive aria-label, since a plain <div> isn't natively focusable or announced as interactive. The keydown handler responds to both Enter and Space. The modal's close button has its own aria-label="Close", and the modal can also be dismissed with the Escape key.

Content Page

Technical Description

content_ST1.html has one <h1> and five <h2> sections, each with a unique id, linked from a sidebar navigation list built with CSS Grid (grid-template-columns: 230px 1fr), with position: sticky on the sidebar so it stays in view while the article scrolls. A media query collapses this to one column below 820px. The "Go to Top" button is a plain anchor link styled with position: fixed — deliberately HTML/CSS only, as required. Its four images are credited in reference 4.

Accessibility

The page keeps a single <h1> followed by five <h2> section headings with no skipped levels. The sidebar links are ordinary anchor links, and each target section has scroll-margin-top so jumping to it doesn't hide the heading under the sticky site header. All four article images have descriptive alt text.

Challenges and Lessons Learned

Getting the hero banner to stretch full-width while the rest of the page stayed centred took some trial and error with the 100vw / negative-margin technique before it worked without introducing a horizontal scrollbar. Making the Gallery's click-to-expand modal keyboard-accessible, not just mouse-hover, meant learning that :hover alone isn't enough for accessibility, and that tabindex plus JavaScript event listeners were needed instead. Coordinating file names and the shared style.css with teammates so nobody's page broke another's was an ongoing exercise in communication as much as code — a small naming mismatch in one shared file could silently break someone else's page.

Accessibility Considerations

Across the template, Home, Gallery and Content pages, accessibility was built in rather than added afterwards. The shared <header>, <nav>, <main> and <footer> use real semantic elements instead of generic <div>s, so the page's structure is announced correctly to screen readers before any content is even read. Every image across all four pages, including the logo, carries descriptive alt text rather than a filename or empty string. The mobile nav toggle keeps its aria-expanded attribute in sync with its open/closed state via JavaScript, so assistive technology always reports the current menu state correctly. On the Gallery page, thumbnails are plain <div>s made keyboard-operable with tabindex="0", role="button" and a descriptive aria-label, since a div isn't natively focusable or announced as interactive; the modal responds to both Enter and Space, has its own aria-label="Close" on the close control, and can also be dismissed with Escape. On the Content page, heading levels never skip (a single <h1> followed by properly nested <h2>s), and every anchor-linked section has scroll-margin-top so jumping to it doesn't hide the heading behind the sticky site header. These choices matter because they mean the site works the same way whether someone is using a mouse, a keyboard, or a screen reader, rather than assuming everyone interacts with it the same way I do.
