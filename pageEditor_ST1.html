Shared Template & Navigation

HTML Structure

template.html defines the structure every other page in the site is built from: a <header> containing the logo (an <img> with descriptive alt text) and a <nav> with links to every main page; a <main> left empty as the slot each student fills with their own content; and a <footer> containing four columns (brand, page links, page-owner links, student details). HTML comments mark the header/nav/footer as "DO NOT MODIFY" so the structure stays consistent once other students copy the file.

CSS Techniques

The header is two-tier: a thin .utility-bar row (SDG badge + accessibility link) sits above a .main-nav row built with Flexbox, where the logo is visually centred between two <nav> groups (.nav-left, .nav-right) using flex: 1 on each group and flex: 0 0 auto on the logo, plus a prominent .nav-cta button at the far right. This structure was inspired by a real-world navigation pattern (a two-tier bar with a centred logo and a call-to-action button), adapted with BrightPath's own page groupings and wording rather than copied. On narrow screens, a media query switches .main-nav to flex-direction: column so the groups stack and centre instead of overlapping. Navigation links use an underline-on-hover style with a .active class each student applies to their own page's link. The footer uses a responsive grid (repeat(auto-fit, minmax(...))) so its four columns collapse cleanly on narrow screens without a media query.

Design Rationale

The template intentionally has as few opinions as possible about page content — only the shared chrome (header/nav/footer) is fixed, everything inside <main> is left to the individual student, which is what the spec's "no structural changes permitted" rule requires while still giving each page room to look different where it should. The colour palette (navy #123A7A, blue #2563EB, sky blue #38BDF8, success green #22C55E) and the Poppins/Inter font pairing were agreed as the team's shared visual identity and implemented centrally in css/global.css so every page inherits them automatically rather than each student picking their own colours.

Home Page

HTML Structure

home.html follows the shared template, then adds a <section class="hero-banner"> for the mission statement, a .cards-grid of three <article class="card"> elements (each with a photo, heading, description, and a checklist), and a .focus-list of anchor links to all four students' Content Pages.

CSS Techniques

The hero banner uses the "full-bleed" trick (width: 100vw; position: relative; left: 50%; margin-left: -50vw;) to break out of the centred <main> container edge to edge, while an inner .hero-inner wrapper keeps the actual content readably constrained. The three cards use CSS Grid with repeat(auto-fit, minmax(280px, 1fr)) so the layout reflows responsively without a manual breakpoint. Page-specific rules (the hero fade-in animation, the card hover-lift, the focus-list hover states) live in an embedded <style> block inside home.html itself, per the coursework's requirement that Home Page CSS be embedded rather than external.

Design Rationale

Cards were used instead of plain paragraphs to keep the "what can I actually do" message scannable. The .focus-list doubles as the "interactive gallery linking to content pages" the spec requires — each item shows a number, a title, and a short description of what that teammate's page covers, with a hover animation to signal it's clickable.

Gallery Page

HTML Structure

Each thumbnail in gallery.html is a <div class="thumb" tabindex="0" role="button" aria-label="..."> carrying data-title, data-desc, and data-img attributes. A hidden <div id="galleryModal"> acts as the template for the extended view and is populated by JavaScript rather than duplicated per photo.

CSS Techniques

Gallery-specific styling lives in its own external stylesheet, css/gallery.css, rather than an embedded block — the spec allows either approach for this page, and a separate file kept it easier to edit independently of the HTML. It defines the thumbnail grid, the modal overlay/positioning, and two extra classes (.alt-color, .alt-font) that are toggled on the modal only, for the user-customisation requirement. The page's JavaScript is similarly extracted into js/gallery.js rather than an inline <script> block, matching the project's overall CSS/JS/images folder separation.

JavaScript Logic

All interactivity is handled with addEventListener rather than CSS :hover, which the spec explicitly requires: click and keydown (Enter/ Space) open the modal and fill it from the clicked thumbnail's data-* attributes; mouseenter/mouseleave/focus/blur toggle a .thumb-hover class for the hover feedback; the modal closes on its close button, a click on the dark overlay, or the Escape key. Two buttons inside the modal toggle .alt-color/.alt-font on .modal-content only, so the customisation never touches the rest of the page.

Design Rationale

Using data-* attributes instead of a separate JS array keeps each thumbnail self-describing directly in the HTML, which made the JavaScript itself shorter (one shared openModal() function reads whichever thumbnail was interacted with, instead of one function per photo). tabindex="0", role="button", and aria-label were added to every thumbnail so keyboard and screen-reader users get the same interaction as mouse users, not just visual hover feedback.

Content Page — "From Passing Grades to Real Skills"

HTML Structure

One <h1> and five <h2> sections, each with a unique id (#exam-treadmill, #skills-gap, #whats-changing, #what-you-can-do, #outcome), linked from a sidebar navigation list via anchor links (href="#section-id"). The page uses a two-column layout: a <nav class="mag-sidebar"> holding the section links and a pinned fact, and a .mag-content column holding the article itself.

CSS Techniques

The layout uses CSS Grid (grid-template-columns: 230px 1fr) to place the sidebar beside the article, with position: sticky on the sidebar so it stays in view while the article scrolls. A media query collapses this to a single column and turns off the sticky positioning below 820px. Pull-quotes use float: right to sit beside the surrounding paragraph text rather than breaking the flow, matching a magazine-style layout. Each section has scroll-margin-top so anchor-jumping doesn't tuck the heading under the sticky site header. The "Go to Top" button is a plain anchor link (href="#top") styled with position: fixed — deliberately no JavaScript, since the spec requires this element to be HTML/CSS only.

Design Rationale

This page went through two full rewrites. The first draft covered a different topic (equitable access to learning resources) in a simpler single-column layout; after review, both the topic and the layout were replaced. The current topic — the gap between exam results and job-readiness — was chosen because it connects to the Home Page's "focus area" card for this page and gives the page a clearer point of view than a general access-and-resources framing. The sidebar/magazine layout was chosen over a flat list of sections to make the page visually distinct from the Gallery and Home pages, and to make the pinned "worth knowing" fact visible throughout the whole read instead of only at the top.

Challenges & Lessons Learned

Placeholder — rewrite in your own words

The draft points below are based on real technical decisions made in the code, but they need to be rewritten as your own account of what was actually difficult and what you actually learned — not left as-is.

  • Getting the hero banner to stretch full-width while the rest of the page stayed centred took some trial and error with the 100vw / negative-margin technique before it worked without introducing a horizontal scrollbar.
  • Making the Gallery's click-to-expand modal keyboard-accessible (not just mouse-hover) meant learning that :hover alone isn't enough for accessibility, and that tabindex + JS event listeners were needed instead.
  • Coordinating file names and the shared style.css with teammates so nobody's page broke another's — agreeing early on which CSS classes were "shared" vs "page-specific" helped avoid conflicts.

Accessibility Considerations

Across all the pages I built, accessibility was addressed with:

  • Meaningful alt text on every image, describing what the image shows rather than just naming the file.
  • Semantic HTML — <header>, <nav>, <main>, <section>, <footer> used for their actual structural meaning rather than generic <div>s throughout.
  • Keyboard access on the Gallery's interactive thumbnails via tabindex="0", role="button", and an Enter/Space key handler identical in effect to a click.
  • Logical heading order on the Content Page (a single <h1> followed by <h2> section headings, never skipping a level).
  • Visible focus/hover feedback so keyboard users get the same interaction cues as mouse users, not just a mouse-only hover effect.

These choices improve usability for screen-reader and keyboard-only users, and also make the pages easier for anyone to use on a slow connection where images may fail to load (the alt text still conveys the content).
