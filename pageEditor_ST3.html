Feedback & Programme Directory Page

Technical Description

The page follows the shared template with a <main> containing three sections: Programme Directory, Reviews Summary, and Feedback Form, each wrapped in a semantic <section> with its own id for internal linking. The Programme Directory uses <article class="program-card"> per item (images credited in reference 1) inside a horizontally-scrollable Flexbox row (display: flex; overflow-x: auto;). The Reviews section uses <article class="review-card"> per review, laid out with CSS Grid (repeat(auto-fit, minmax(280px, 1fr))). The Feedback Form uses <fieldset>/<legend> for field grouping (reference 3), with text, email, tel, date, datetime-local inputs, a textarea, and a <datalist> (reference 3) for programme suggestions. js/feedback.js provides a live character counter on the message textarea, submit-time validation that checks name presence, an email-format regex, and comment presence, writing errors into each field's .field-error span, and a submission confirmation message.

Accessibility

Every form field is wrapped in its own <label>, and the two field groups use <fieldset>/<legend> so a screen reader announces which group a field belongs to. Validation errors appear in a .field-error span immediately next to the field that failed. The submission confirmation message uses aria-live="polite". Every programme and review image has descriptive alt text.

Team Page

Technical Description

The page uses the shared template plus a team introduction paragraph and a .team-grid container built with CSS Grid (repeat(auto-fit, minmax(230px, 1fr))). Each team member is a <div class="team-card"> with role="group", tabindex="0", and a descriptive aria-label, containing a .team-photo-wrap image (real portraits of all four team members), a .team-basic name/role block, and a .team-details block with pages built, a summary, and LinkedIn/GitHub links for every member. All styling is embedded in a <style> block: each photo has a fixed aspect-ratio of 1/1 and a grayscale filter that clears to full colour on hover/focus, and the details panel is hidden via max-height: 0; opacity: 0; and revealed with a transition on :hover, :focus, and :focus-within.

Accessibility

Each card's tabindex="0" means the same expand-on-hover reveal also triggers on keyboard focus, via :focus-within, so keyboard users get the same "more details" behaviour as mouse users. role="group" plus a descriptive aria-label gives each card an accessible name beyond just the visible text. Every portrait has descriptive alt text.

Content Page: Community Learning Programs

Technical Description

The page uses the shared template plus a .content-wrapper containing a hero section, a sticky internal navigation bar (position: sticky; top: 90px;), six content sections, and a "Go to Top" button. Five images (credited in reference 1) illustrate real community-learning initiatives referenced by name, the British Council EDGE programme and UNICEF's Reading for Pleasure initiative. All styling is embedded in a <style> block: the hero uses a gradient background with large rounded corners, and each content section is a rounded white card with a soft shadow.

Accessibility

The page keeps a single <h1> (in the hero) followed by <h2> section headings and occasional <h3>s for named programmes, with no skipped levels. All five images have descriptive alt text, and the internal navigation uses ordinary anchor links, reachable without a mouse.

Challenges and Lessons Learned

Implementing the live character counter meant properly understanding DOM manipulation and event listeners rather than just updating text on submit. Getting the Team page's expand/collapse details to animate smoothly required learning proper CSS transition technique using max-height and opacity together, since max-height alone snaps rather than eases. Making the Content Page's sticky internal navigation stay visible while scrolling, without overlapping the sticky site header, took some care with the top offset and z-index. Building custom form validation with clear, field-specific error messages rather than one generic alert meaningfully improved how I think about JavaScript-driven UX.

Accessibility Considerations

Across the Feedback & Programme Directory, Team and Content pages, accessibility was considered at both the form and interaction level. Every field in the Feedback form is wrapped in its own <label>, and the two field groups use <fieldset>/<legend> so a screen reader announces which group a field belongs to before reading its individual label. Validation errors appear in a .field-error span immediately next to the field that failed, rather than in one generic alert disconnected from the problem, and the submission confirmation message uses aria-live="polite" so it's announced as soon as it appears. On the Team page, each card's tabindex="0" means the same expand-on-hover reveal also triggers on keyboard focus via :focus-within, so keyboard users get the same "more details" behaviour as mouse users, and role="group" plus a descriptive aria-label gives each card an accessible name beyond its visible text. On the Content page, heading levels never skip — a single <h1> followed by <h2>s and occasional <h3>s for named programmes — and the internal navigation uses ordinary anchor links, reachable without a mouse. Every image across all three pages carries descriptive alt text. The common thread is that interactive behaviour (form validation, hover reveals) is always paired with an equivalent that works on focus and is announced to assistive technology, not just visible to a sighted mouse user.
