Feedback & Programme Directory

HTML Structure

A scrollable row of <article class="program-card"> elements, a ratings summary, three <article class="review-card"> elements, and a <form> split into two <fieldset> groups ("Personal Details" and "Feedback Details") as the spec requires. The form includes text, email, tel, date, datetime-local inputs, a textarea with maxlength, and a <datalist> for searchable programme suggestions.

CSS Techniques

The programme directory uses overflow-x: auto on a flex row so it scrolls horizontally without taking over the page, per the spec's "scrollable, not full-page" requirement. Form fields use a consistent label-above-input pattern with a visible focus outline.

JavaScript Logic

A live character counter updates on every input event on the comment textarea. Form submission is intercepted with preventDefault(), each required field is checked individually (name presence, an email-format regex, comment presence), and errors are written into a dedicated <span class="field-error"> next to each field rather than a single generic alert. A success message appears and the form resets once validation passes.

Team Page

HTML Structure

Four <div class="team-member" tabindex="0"> blocks, each containing an initials avatar, a name, a role, and a .team-details block with more information.

CSS Techniques

Since real photos weren't available for every team member, each card uses a CSS-styled initials avatar instead — an honest fallback the spec explicitly allows ("a thumbnail image, or icon if an image is not available"). The detail panel is hidden by default via max-height: 0; overflow: hidden; and revealed on :hover, :focus, and :focus-within with a transition, so it works for keyboard users tabbing onto the card, not just mouse users.

Design Rationale

tabindex="0" was added to each card specifically because the expand-on-hover interaction needed a keyboard equivalent — the spec calls this out directly as an accessibility requirement for this page.

Challenges & Lessons Learned

Placeholder — rewrite in your own words
  • Getting the horizontally-scrolling programme row to work smoothly on both desktop (mouse wheel/drag) and mobile (touch swipe) without extra JavaScript.
  • Deciding what counted as "enough" custom JavaScript validation on top of the HTML required/type="email" attributes, per the spec's requirement that JavaScript add genuinely extra feedback.
  • Making the Team page's hover-reveal panel also work for keyboard users, not just mouse hover.

Accessibility Considerations

  • Alt text on every programme/review image.
  • Labelled form fields — every input is wrapped in a <label>, and related fields are grouped with <fieldset>/<legend>.
  • Error messages placed next to their field, not just at the top of the form, so screen-reader users encounter them in context.
  • Keyboard-accessible Team cards via tabindex="0" and :focus-within so the detail panel opens without a mouse.
