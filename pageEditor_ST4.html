User Profile

Technical Description

No HTML form at all, per the spec — instead, a three-card step-overview (numbered badges 1–3) explaining what each step covers, a control area (Start/Continue, Reset, and three "revisit step" buttons), a progress bar, and an empty <div id="profileOutput"> that JavaScript populates as the user answers prompts. The progress bar is a nested-div track/fill pattern where the fill's width percentage is set directly by JavaScript, with a 0%/50%/100% scale printed underneath it. Step and prompt definitions live in a STEPS array of objects (three steps, three prompts each). A runStep() function loops through a step's prompts calling window.prompt() (reference 3) for each; a blank response or Cancel is stored as null and treated as "skipped". renderProfile() clears and rebuilds the output area from profileData using createElement/textContent (DOM manipulation, not innerHTML string-building), and updateProgress() recalculates the percentage from how many non-null answers exist. Any step can be revisited via its button at any time, and each step button gains a "step-complete" class once visited so the border colour shows at a glance which steps still need attention.

Accessibility

Progress is shown as visible text, a percentage and prompt count, alongside the progress bar's colour fill, not through colour alone. Using the browser's native window.prompt() instead of a custom on-page dialog means every prompt is a standard OS-level control that's already fully keyboard-operable and screen-reader-compatible.

Sitemap

Technical Description

A single inline <svg viewBox="0 0 1200 480"> containing one <a href="..."> element per page, each wrapping a <rect> and <text>, plus a <title> for its accessible name, and a set of <line> elements showing the parent/child relationships between Home and every other page. viewBox combined with width: 100%; height: auto; in CSS makes the whole diagram scale proportionally at any screen size without JavaScript. Hover/focus states are handled with plain CSS selectors since SVG <a> elements with an href are natively part of the tab order in modern browsers (reference 3). Every page in the site is represented as its own node rather than grouping them, so the diagram doubles as an accurate, literal map of the site. A colour-coded legend above the diagram explains what each node style means, and a plain-text "Quick Links" navigation list below it repeats every page as an ordinary hyperlink, as a fallback for anyone who finds a large hand-built SVG diagram awkward to navigate.

Accessibility

Each node's <a href> wraps a <title> element giving it an accessible name beyond just its visible label text. Because SVG links with an href are natively part of the tab order in modern browsers, the whole diagram is keyboard-navigable without any extra tabindex or JavaScript, and each node gets a visible :focus outline. The plain-text Quick Links list underneath gives the same 12 destinations again as ordinary anchor links, so the page's core navigation doesn't depend on successfully parsing the diagram at all.

Content Page: Lifelong Skills for the Future

Technical Description

One <h1> and five <h2> sections, linked from a simple top navigation list, built on the same shared css/content.css as Students 2 and 3, with a handful of page-scoped "st4-" prefixed classes added to that same shared file for the two-column hero, the pull-stat callout, section intro lines, image captions, and the photo-credit line specific to this page. The hero uses a Flexbox two-column layout (text on one side, a supporting photo on the other) so the header area doesn't leave an empty gap on wide screens, and collapses to a single column below 860px. Each of the five sections is a bordered card with a numbered circular badge (01–05, aria-hidden since the number is decorative) next to its <h2>, and each section's photo is wrapped in a <figure>/<figcaption> pair rather than a plain <img>+<span>, sized to its own natural aspect ratio (capped at a max-height) and centred, so nothing gets cropped or letterboxed regardless of the source photo's proportions. Its six images (one hero photo plus one per section) are credited in reference 1. The topic focuses on the skills a fixed curriculum doesn't teach directly, connecting to the Home Page's fourth focus-area card. A fifth section, "The Real Barrier", sits between the problem and the solutions to address why students don't already use the opportunities that exist — awareness and confidence, not ability — before moving on to what a reader can actually do about it.

Accessibility

All six article images (hero plus five section photos) have descriptive alt text describing the actual scene rather than a generic label, and the single <h1> plus five <h2> sections keep the heading outline simple and predictable. Each section's <figcaption> gives its image a real accessible caption rather than a purely visual label. The top navigation list uses ordinary anchor links, reachable and activatable without a mouse.

Challenges and Lessons Learned

Deciding how to show a "skipped" prompt took a bit of thinking — I didn't want it to read like an error, so a skipped answer shows up as an italic "Skipped" label in the profile output rather than being left blank or hidden entirely. Positioning all 12 nodes and their connecting lines in the sitemap's SVG by hand took a fair amount of trial and error to stop the labels overlapping, especially once the four content-page nodes went on their own row underneath the six main pages. Making sure the sitemap was genuinely usable with just a keyboard, not only a mouse, meant actually testing the tab order rather than assuming SVG <a> links would behave exactly like normal HTML links. When I went back to expand the Content Page, the balance was making it feel complete without turning it into a full magazine-style layout like Student 1's — it still needed to share the same simpler template as Students 2 and 3, just with a bit more depth.

Accessibility Considerations

Across the User Profile, Sitemap and Content pages, accessibility was mostly about giving every interactive element a text-based equivalent alongside its visual one. On the User Profile page, using the browser's native window.prompt() instead of a custom on-page dialog means every prompt is a standard OS-level control that's already fully keyboard-operable and screen-reader-compatible without any extra work, and progress is shown as visible text (a percentage and prompt count) alongside the progress bar's colour fill, not through colour alone. On the Sitemap page, each node's <a href> wraps a <title> element giving it an accessible name beyond its visible label text; because SVG links with an href are natively part of the tab order in modern browsers, the whole diagram is keyboard-navigable without any extra tabindex or JavaScript, and each node gets a visible :focus outline; the plain-text "Quick Links" list underneath repeats every destination as an ordinary anchor link, as a fallback for anyone who finds a large hand-built SVG diagram awkward to navigate. On the Content page, the single <h1> plus five <h2> sections keep the heading outline simple and predictable, and every image carries descriptive alt text describing the actual scene rather than a generic label. Taken together, these choices mean none of the three pages' more unusual interaction patterns (prompt-based forms, an inline SVG diagram) depend on sight or a mouse to be usable.

Compliance with JANET Regulations

The User Profile page collects information only into browser memory via prompt() — nothing is transmitted, stored server-side, or shared, so there is no personal-data handling risk to manage, in keeping with JANET's Acceptable Use Policy around not introducing unnecessary data-collection or security exposure on JANET-connected networks.
