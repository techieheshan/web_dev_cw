User Profile

HTML Structure

No HTML form at all, per the spec — instead, a control area (Start/Continue, Reset, and three "revisit step" buttons), a progress bar, and an empty <div id="profileOutput"> that JavaScript populates as the user answers prompts.

CSS Techniques

The progress bar is a nested-div track/fill pattern where the fill's width percentage is set directly by JavaScript. Step buttons gain a .step-complete class once visited, changing their border colour so the user can see at a glance which steps still need attention.

JavaScript Logic

Step and prompt definitions live in a STEPS array of objects (three steps, three prompts each). A runStep() function loops through a step's prompts calling window.prompt() for each; a blank response or Cancel is stored as null and treated as "skipped" rather than as an error. profileData is a plain object keyed by step number that accumulates answers across visits. renderProfile() clears and rebuilds the output area from profileData using createElement/textContent (DOM manipulation, not innerHTML string-building, for the actual field values) every time it's called, and updateProgress() recalculates the percentage from how many non-null answers exist. Any step can be revisited via its button at any time, which lets a user go back and fill in something they skipped earlier — both the profile output and the progress bar update immediately.

Sitemap

HTML Structure

A single inline <svg viewBox="0 0 1200 470"> containing one <a href="..."> element per page (each wrapping a <rect> and <text>, plus a <title> for its accessible name), and a set of <line> elements showing the parent/child relationships between Home and every other page.

CSS Techniques

viewBox combined with width: 100%; height: auto; in CSS makes the whole diagram scale proportionally at any screen size without JavaScript. Hover/focus states are handled with plain CSS selectors (.site-node:hover .node-box) since SVG <a> elements with an href are natively part of the tab order in modern browsers.

Design Rationale

Every page in the site is represented as its own node rather than grouping them, so the diagram doubles as an accurate, literal map of the site rather than a simplified illustration.

Challenges & Lessons Learned

  • Working out how to represent "skipped" prompts clearly in both the profile output and the progress percentage, instead of just ignoring them silently.
  • Positioning 12 SVG nodes and their connecting lines by hand so the diagram stayed readable without overlapping labels.
  • Making sure the SVG sitemap was still fully keyboard-navigable, not just clickable with a mouse.

Accessibility Considerations

  • Text alternatives on every SVG node via a <title> element giving each link an accessible name beyond its visible label.
  • Keyboard navigation through the sitemap using native SVG <a href> tab order, with a visible :focus outline.
  • Live-updating progress on the Profile page is reflected in visible text (the percentage and prompt count), not colour alone.
